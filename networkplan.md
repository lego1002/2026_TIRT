-Fi 路由器:發出 SSID、給連上來的裝置發 IP(DHCP)、幫它們互相轉發封包。關鍵在於:

  Pi ──Wi-Fi──► 手機(當路由器) ◄──Wi-Fi── 筆電
                  │
                                  └╌╌ 4G/5G(這段跟機器人完全無關)

                                    Pi → 筆電的 ROS 2 資料(/scan、/map、/tf)只在熱點的區網內繞,根本不會經過手機的行動網路。 所以你擔心的「手機連線不穩」不影響 —— 就算手機完全沒收訊、開飛航模式再開熱點,Pi
                                      和筆電照樣互通,RViz 照樣有畫面。行動網路唯一的用途是讓 Tailscale 能動;沒有網際網路時,改用 ssh lego@lego4g.local(mDNS,你的 Pi 已裝好 avahi)直接連區網 IP 就行。

                                        這也是為什麼它解決了你的核心痛點:你不再依賴場地有沒有網路 —— 網路是你帶去的,而且 Pi 和筆電「保證」在同一個網段,因為它們連的是同一台你控制的「路由器」。

                                          大家常用嗎?

                                            是,這是機器人比賽的標準做法。RoboCup、各種校際賽的隊伍幾乎都自帶網路(手機熱點、筆電熱點、或乾脆帶一台小路由器),原因就是下一段講的場地網路不可靠。你的資料量也完全不是問題:lidar
                                              /scan 10 Hz、/map 幾秒一張、/tf 幾十 Hz,加起來遠低於 2.4 GHz 熱點的頻寬,區網延遲只有幾毫秒。

                                                手機熱點 vs 筆電熱點

                                                  ┌────────────────────────────┬────────────────────────────────────────────────────────────────┬────────────────────────────────────────┐
                                                    │                            │                            手機熱點                            │                筆電熱點                │
                                                      ├────────────────────────────┼────────────────────────────────────────────────────────────────┼────────────────────────────────────────┤
                                                        │ Pi↔筆電路徑                │ 都是客戶端,經手機轉發                                          │ Pi 直連筆電(筆電就是 AP)               │
                                                          ├────────────────────────────┼────────────────────────────────────────────────────────────────┼────────────────────────────────────────┤
                                                            │ 有網際網路(Tailscale 可用) │ ✅ 有 4G 就有                                                  │ ❌ 通常沒有(單一無線網卡)              │
                                                              ├────────────────────────────┼────────────────────────────────────────────────────────────────┼────────────────────────────────────────┤
                                                                │ AP 隔離風險                │ ⚠️  部分手機會擋客戶端互連 → SSH via tailscale 還通,但 DDS 會死 │ ✅ 無此問題                            │
                                                                  ├────────────────────────────┼────────────────────────────────────────────────────────────────┼────────────────────────────────────────┤
                                                                    │ 開啟方式                   │ 系統設定一鍵                                                   │ nmcli device wifi hotspot ... 一行指令 │
                                                                      └────────────────────────────┴────────────────────────────────────────────────────────────────┴────────────────────────────────────────┘

                                                                        「AP 隔離(client isolation)」是唯一要小心的坑:有些手機的熱點會禁止連上來的裝置互相通訊(安全考量)。所以手機熱點要在賽前實測一次 —— 兩台都連上後 ping 對方、ros2 topic echo /scan
                                                                          有資料就沒問題。我的建議是兩個都設進 Pi:平常用手機熱點(順便有網路、Tailscale 照常用),筆電熱點當備援(百分之百可靠,但沒網際網路)。

                                                                            大會提供網路的話?

                                                                              還是用自己的,別猶豫。 公共/場地 Wi-Fi 對 ROS 2 是地雷區:

                                                                                1. 幾十隊同時用,壅塞、掉包,你的 /map 傳輸會卡。
                                                                                  2. 公共 AP 幾乎都開 AP 隔離 —— 你們兩台各自連得上網,但互相 ping 不到,DDS 直接失效(症狀就是你熟悉的「topic list 看得到、RViz 一片空白」)。
                                                                                    3. multicast 常被擋 —— DDS 的節點探索靠 multicast,被擋就要手動設 ROS_STATIC_PEERS,現場除錯很痛苦。
                                                                                      4. 可能有登入頁(captive portal),Pi 是 headless 根本沒辦法點「同意」。

                                                                                        場地網路頂多拿來讓手機/筆電上網查資料;機器人的控制鏈路一律走自己的熱點,這樣你在宿舍測好的東西,到了賽場行為一模一樣 —— 這種「環境不變性」在比賽當天比什麼都值錢。
